Overview

This is the canonical sBTC fungible token contract deployed on Stacks mainnet. It implements the SIP-010 trait and adds a dual-token model: sbtc-token (liquid, transferable) and sbtc-token-locked (non-transferable, used for protocol-level locking). All privileged operations (mint, burn, lock, unlock, metadata changes) are gated via .sbtc-registry protocol caller checks using a contract-flag byte.

The contract is remarkably compact for its importance — ~157 lines governing the entire sBTC supply on Stacks. The code is clean, well-structured, and delegates access control to the external registry. No maps or data structures beyond three metadata variables.

Architecture

  • Dual fungible tokens: sbtc-token (liquid) and sbtc-token-locked (locked) — locking burns liquid tokens and mints locked tokens, unlocking reverses the process
  • Protocol functions: All mint/burn/lock/unlock operations require .sbtc-registry is-protocol-caller authorization via contract-flag
  • SIP-010 compliance: Standard transfer, get-name, get-symbol, get-decimals, get-balance, get-total-supply, get-token-uri
  • Batch operations: transfer-many (up to 200 transfers) and protocol-mint-many (up to 200 mints) for gas-efficient bulk operations
  • Balance model: get-balance returns the sum of liquid + locked; get-balance-available and get-balance-locked return individual components

Findings

MEDIUM

M-01: transfer-many allows any caller to transfer from arbitrary senders

Location: transfer-many (line 84) → transfer (line 119)

Description: The transfer-many function accepts a list of transfers where each item specifies its own sender. Each individual transfer is delegated to transfer, which correctly checks:

(asserts! (or (is-eq tx-sender sender) (is-eq contract-caller sender)) ERR_NOT_OWNER)

This means the tx-sender must equal each item's sender, OR the contract-caller must equal the sender. For direct calls, tx-sender == contract-caller, so users can only transfer their own tokens. However, when called via a contract, the contract-caller is the intermediary contract. If a user approves a contract to act on their behalf (e.g., a DEX router), that contract can use transfer-many to batch transfers from the user. This is by design and consistent with SIP-010 conventions.

The concern is that the sender field in each tuple could differ — a malicious contract could mix senders in a single transfer-many call. However, since transfer validates each item individually, the contract-caller must equal every specified sender, which means all items must have the same sender (the calling contract itself). This limits the actual attack surface.

Impact: No direct exploit exists because the per-item transfer check prevents cross-sender abuse. However, the tuple design is misleading — it appears to allow heterogeneous sender batches but actually constrains all senders to be tx-sender or contract-caller. A contract acting as contract-caller can batch-transfer its own held tokens, which is expected behavior. The real risk is that a user interacting with a malicious contract inadvertently authorizes that contract as contract-caller to transfer their tokens via SIP-010's standard auth model.

Recommendation: This is inherent to SIP-010's contract-caller auth model and not specific to this contract. Users should exercise caution when interacting with unverified contracts. Consider documenting that transfer-many enforces per-item sender validation.

LOW

L-01: protocol-mint-many does not propagate individual mint failures

Location: protocol-mint-many (line 76) → protocol-mint-many-iter (line 72)

Description: The batch mint function uses map over the recipients list:

(define-public (protocol-mint-many (recipients (list 200 {amount: uint, recipient: principal})) (contract-flag (buff 1)))
  (begin
    (try! (contract-call? .sbtc-registry is-protocol-caller contract-flag contract-caller))
    (ok (map protocol-mint-many-iter recipients))
  )
)

The map function in Clarity does not short-circuit on error. The iterator protocol-mint-many-iter returns (response bool uint) from ft-mint?. If one mint fails (e.g., overflow), the map collects the error in the result list but continues processing. The outer function returns (ok (list ...)) containing a mix of (ok true) and (err ...) values.

Impact: A partial failure in a batch mint is silently absorbed into the result list. The caller must inspect each element to detect failures. In practice, ft-mint? only fails on uint overflow (supply exceeding u340282366920938463463374607431768211455), which is unrealistic for sBTC. However, the pattern diverges from transfer-many, which uses fold with early termination on error.

Recommendation: Consider using fold with unwrap! (like transfer-many-iter) for consistent error-handling behavior across batch operations. Alternatively, document that callers should check the result list for individual failures.

LOW

L-02: transfer-many error indexing starts at u1000, limiting batch size diagnostics

Location: transfer-many-iter (line 93)

Description: When an individual transfer within a batch fails, the error code is computed as:

(err (+ ERR_TRANSFER_INDEX_PREFIX index))

Where ERR_TRANSFER_INDEX_PREFIX = u1000. This means error u1000 = first transfer failed, u1001 = second, etc. This is a well-designed pattern for debugging batch failures. However, the prefix u1000 overlaps with potential error codes from the broader sBTC protocol (e.g., if sbtc-registry returns error codes in the thousands).

Impact: Minimal. The error code space is large enough that collisions are unlikely in practice. The u1000+ range clearly indicates a batch index error, and the index is easily extracted by subtracting the prefix.

Recommendation: No change needed. The pattern is clean and practical.

INFO

I-01: Mutable token metadata (name, symbol, URI) controlled by protocol

Location: protocol-set-name, protocol-set-symbol, protocol-set-token-uri

Description: Token name, symbol, and URI are stored as data-var rather than constants, and can be changed by any protocol-authorized caller. While this provides flexibility for rebranding or URI migration, it means the token's identity is not immutable.

Impact: A compromised protocol caller could change the token name/symbol to something misleading. This is mitigated by the sbtc-registry access control — only authorized protocol contracts can make changes. Wallets and explorers that cache metadata would show stale values until refreshed.

Recommendation: This is a deliberate design choice providing upgrade flexibility. The registry access control is appropriate. Off-chain systems should subscribe to contract events or periodically refresh metadata.

INFO

I-02: get-balance returns sum of liquid and locked — potential UX confusion

Location: get-balance (line 140)

Description: The SIP-010 standard get-balance function returns:

(ok (+ (ft-get-balance sbtc-token who) (ft-get-balance sbtc-token-locked who)))

This includes locked tokens in the reported balance. Wallets and DeFi protocols querying get-balance will see a higher balance than what the user can actually transfer. The contract provides get-balance-available and get-balance-locked for granular queries, but these are non-standard extensions that third-party integrations may not know to use.

Impact: Users may see inflated sBTC balances in wallets that only use get-balance. Attempting to transfer the full reported balance would fail if some tokens are locked. Well-integrated wallets (Leather, Xverse) are likely aware of the available/locked distinction.

Recommendation: This is a reasonable design decision — the total balance is the most accurate representation of holdings. Wallet integrations should use get-balance-available for transfer UIs. Consider adding documentation or SIP-010 metadata pointing to the extended balance functions.

Positive Observations

  • Minimal attack surface: At ~157 lines, this is one of the most compact critical contracts on Stacks. Less code = fewer bugs. No maps, no complex state transitions, no reentrancy vectors.
  • Clean access control delegation: All privileged operations consistently delegate to .sbtc-registry is-protocol-caller with a contract-flag byte, enabling fine-grained permissioning per operation type without hardcoding specific callers.
  • Dual-token lock model: Using separate fungible tokens for liquid and locked balances is elegant — it leverages Clarity's built-in FT accounting rather than maintaining a parallel balance map, eliminating an entire class of accounting bugs.
  • Proper sender validation: transfer checks both tx-sender and contract-caller against the sender parameter, correctly implementing SIP-010 authorization semantics.
  • Informative batch errors: transfer-many uses an index-prefixed error scheme (u1000+) that pinpoints exactly which transfer in a batch failed — a thoughtful debugging aid.
  • Memo support: Transfer functions properly support optional 34-byte memos with print statements, enabling off-chain indexing and cross-chain message tagging.
